1. Introduction

JMAP ("The JSON Meta Application Protocol (JMAP)" [RFC8620]) is a generic protocol for synchronising data, such as mail, calendars, or contacts, between a client and a server. It is optimised for mobile and web environments, and it provides a consistent interface to different data types.Â¶

JMAP for Mail ("The JSON Meta Application Protocol (JMAP) for Mail" [RFC8621]) specifies a data model for synchronising email data with a server using JMAP. Clients can use this to efficiently search, access, organise, and send messages.Â¶

Message Disposition Notifications (MDNs) are defined in [RFC8098] and are used as "read receipts", "acknowledgements", or "receipt notifications".Â¶

A client can come across MDNs in different ways:Â¶

  1. When receiving an email message, an MDN can be sent to the sender. This specification defines an "MDN/send" method to cover this case.Â¶
  2. When sending an email message, an MDN can be requested. This must be done with the help of a header field, as already specified by [RFC8098]; the header field can already be handled by guidance in [RFC8621].Â¶
  3. When receiving an MDN, the MDN could be related to an existing sent message. This is already covered by [RFC8621] in the EmailSubmission object. A client might want to display detailed information about a received MDN. This specification defines an "MDN/parse" method to cover this case.Â¶

The same terminology is used in this document as in the core JMAP specification.Â¶

Because keywords are case insensitive in IMAP but case sensitive in JMAP, the $mdnsent keyword MUST always be used in lowercase.Â¶

1.3. Addition to the Capabilities Object

Capabilities are announced as part of the standard JMAP Session resource; see [RFC8620], Section 2. This defines a new capability, "urn:ietf:params:jmap:mdn".Â¶

The capability "urn:ietf:params:jmap:mdn" being present in the "accountCapabilities" property of an account represents support for the "MDN" data type, parsing MDNs via the "MDN/parse" method, and creating and sending MDN messages via the "MDN/send" method. Servers that include the capability in one or more "accountCapabilities" properties MUST also include the property in the "capabilities" property.Â¶

The value of this "urn:ietf:params:jmap:mdn" property is an empty object both in the account's "accountCapabilities" property and in the "capabilities" property.Â¶

2. MDN

An MDN object has the following properties:Â¶

  • forEmailId: Id|nullÂ¶

    The Email id of the received message to which this MDN is related. This property MUST NOT be null for "MDN/send" but MAY be null in the response from the "MDN/parse" method.Â¶

  • subject: String|nullÂ¶

    The subject used as "Subject" header field for this MDN.Â¶

  • textBody: String|nullÂ¶

    The human-readable part of the MDN, as plain text.Â¶

  • includeOriginalMessage: Boolean (default: false)Â¶

    If true, the content of the original message will appear in the third component of the multipart/report generated for the MDN. See [RFC8098] for details and security considerations.Â¶

  • reportingUA: String|nullÂ¶

    The name of the Mail User Agent (MUA) creating this MDN. It is used to build the MDN report part of the MDN. Note that a null value may have better privacy properties.Â¶

  • disposition: DispositionÂ¶

    The object containing the diverse MDN disposition options.Â¶

  • mdnGateway: String|null (server-set)Â¶

    The name of the gateway or Message Transfer Agent (MTA) that translated a foreign (non-Internet) message disposition notification into this MDN.Â¶

  • originalRecipient: String|null (server-set)Â¶

    The original recipient address as specified by the sender of the message for which the MDN is being issued.Â¶

  • finalRecipient: String|nullÂ¶

    The recipient for which the MDN is being issued. If set, it overrides the value that would be calculated by the server from the Identity defined in the "MDN/send" method, unless explicitly set by the client.Â¶

  • originalMessageId: String|null (server-set)Â¶

    The "Message-ID" header field [RFC5322] (not the JMAP id) of the message for which the MDN is being issued.Â¶

  • error: String[]|null (server-set)Â¶

    Additional information in the form of text messages when the "error" disposition modifier appears.Â¶

  • extensionFields: String[String]|nullÂ¶

    The object where keys are extension-field names, and values are extension-field values (see [RFC8098], Section 3.3).Â¶

A Disposition object has the following properties:Â¶

  • actionMode: StringÂ¶

    This MUST be one of the following strings: manual-action / automatic-actionÂ¶

  • sendingMode: StringÂ¶

    This MUST be one of the following strings: mdn-sent-manually / mdn-sent-automaticallyÂ¶

  • type: StringÂ¶

    This MUST be one of the following strings: deleted / dispatched / displayed / processedÂ¶

See [RFC8098] for the exact meaning of these different fields. These fields are defined as case insensitive in [RFC8098] but are case sensitive in this RFC and MUST be converted to lowercase by "MDN/parse".Â¶

2.1. MDN/send

The "MDN/send" method sends a message in the style of [RFC5322] from an MDN object. When calling this method, the "using" property of the Request object MUST contain the capabilities "urn:ietf:params:jmap:mdn" and "urn:ietf:params:jmap:mail"; the latter because of the implicit call to "Email/set" and the use of Identity objects, which is described below. The method takes the following arguments:Â¶

  • accountId: IdÂ¶

    The id of the account to use.Â¶

  • identityId: IdÂ¶

    The id of the Identity to associate with these MDNs. The server will use this identity to define the sender of the MDNs and to set the "finalRecipient" field.Â¶

  • send: Id[MDN]Â¶

    A map of the creation id (client specified) to MDN objects.Â¶

  • onSuccessUpdateEmail: Id[PatchObject]|nullÂ¶

    A map of the id to an object containing properties to update on the Email object referenced by the "MDN/send" if the sending succeeds. This will always be a backward reference to the creation id (see the example below in Section 3.1).Â¶

The response has the following arguments:Â¶

  • accountId: IdÂ¶

    The id of the account used for the call.Â¶

  • sent: Id[MDN]|nullÂ¶

    A map of the creation id to an MDN containing any properties that were not set by the client. This includes any properties that were omitted by the client and thus set to a default by the server. This argument is null if no MDN objects were successfully sent.Â¶

  • notSent: Id[SetError]|nullÂ¶

    A map of the creation id to a SetError object for each record that failed to be sent or null if all successful.Â¶

In this context, the existing SetError types defined in [RFC8620] and [RFC8621] are interpreted as follows:Â¶

notFound:
The reference "forEmailId" cannot be found or has no valid "Disposition-Notification-To" header field.Â¶
forbidden:
"MDN/send" would violate an Access Control List (ACL) or other permissions policy.Â¶
forbiddenFrom:
The user is not allowed to use the given "finalRecipient" property.Â¶
overQuota:
"MDN/send" would exceed a server-defined limit on the number or total size of sent MDNs. It could include limitations on sent messages.Â¶
tooLarge:
"MDN/send" would result in an MDN that exceeds a server-defined limit for the maximum size of an MDN or more generally, on email message.Â¶
rateLimit:
Too many MDNs or email messages have been created recently, and a server-defined rate limit has been reached. It may work if tried again later.Â¶
invalidProperties:
The record given is invalid in some way.Â¶

The following is a new SetError:Â¶

mdnAlreadySent:
The message has the $mdnsent keyword already set.Â¶

If the "accountId" or "identityId" given cannot be found, the method call is rejected with an invalidArguments error.Â¶

The client MUST NOT issue an "MDN/send" request if the message has the $mdnsent keyword set.Â¶

When sending the MDN, the server is in charge of generating the "originalRecipient" and "originalMessageId" fields according to [RFC8098]. "finalRecipient" will also generally be generated by the server based on the provided identity, but if specified by the client and allowed (see Section 5), the server will use the client-provided value.Â¶

The client is expected to explicitly update each "Email" for which an "MDN/send" has been invoked in order to set the $mdnsent keyword on these messages. To ensure that, the server MUST reject an "MDN/send" that does not result in setting the keyword $mdnsent. Thus, the server MUST check that the "onSuccessUpdateEmail" property of the method is correctly set to update this keyword.Â¶

2.2. MDN/parse

This method allows a client to parse blobs as messages in the style of [RFC5322] to get MDN objects. This can be used to parse and get detailed information about blobs referenced in the "mdnBlobIds" of the EmailSubmission object or any email message the client could expect to be an MDN.Â¶

The "forEmailId" property can be null or missing if the "originalMessageId" property is missing or does not refer to an existing message or if the server cannot efficiently calculate the related message (for example, if several messages get the same "Message-ID" header field).Â¶

The "MDN/parse" method takes the following arguments:Â¶

  • accountId: IdÂ¶

    The id of the account to use.Â¶

  • blobIds: Id[]Â¶

    The ids of the blobs to parse.Â¶

The response has the following arguments:Â¶

  • accountId: IdÂ¶

    The id of the account used for the call.Â¶

  • parsed: Id[MDN]|nullÂ¶

    A map of the blob id to a parsed MDN representation for each successfully parsed blob or null if none.Â¶

  • notParsable: Id[]|nullÂ¶

    A list of ids given that corresponds to blobs that could not be parsed as MDNs or null if none.Â¶

  • notFound: Id[]|nullÂ¶

    A list of blob ids given that could not be found or null if none.Â¶

The following additional errors may be returned instead of the "MDN/parse" response:Â¶

requestTooLarge:
The number of ids requested by the client exceeds the maximum number the server is willing to process in a single method call.Â¶
invalidArguments:
If the given "accountId" cannot be found, the MDN parsing is rejected with an invalidArguments error.Â¶
